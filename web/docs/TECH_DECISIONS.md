## TD-001 — Frontend Language: TypeScript → JavaScript (JSDoc)

**Decision**: Migrate frontend application code from TypeScript to JavaScript with JSDoc annotations.  
**Status**: Approved  
**Date**: 2026-04-07

### Context
The project is AI-agent-assisted development. The primary coding agent has identified limitations in generating production-quality TypeScript at the scale this project requires.

### Problem with TypeScript at this scale
1. **Error handling verbosity** — Typed error recovery, discriminated unions, and typed async flows multiply code size significantly per feature. A single API integration can grow from ~5 lines to 30+ with proper TS patterns.
2. **Agent drift toward `any`** — Under complexity pressure, the agent resolves compiler errors using `any`, `as unknown`, or type assertions. This produces TypeScript overhead with zero type safety benefit.
3. **Refactor cascade** — The feature set is incomplete and actively being scoped. Every schema or contract change in TS cascades through generics, interfaces, and imports across the monorepo — generating noise and rework.
4. **Exponential overhead** — The planned feature set (AI vision pipeline, offline sync, supply chain, GST engine, WhatsApp gateway) is large. TypeScript boilerplate compounds with each addition.

### Decision
- **Frontend app code** (`/apps/desktop`, `/apps/admin-hub`, `/apps/marketplace`): **JavaScript (JSX)** with JSDoc for editor intellisense
- **GST calculation engine** (`/packages/accounting`): **TypeScript retained** — compliance-critical, calculation errors have legal implications
- **Supabase DB types**: **TypeScript retained** — auto-generated by Supabase CLI, zero maintenance cost, provides typed DB client across all apps

### JSDoc Approach
JSDoc annotations give 80% of TypeScript's editor benefits (autocomplete, hover types, basic safety) with none of the compiler overhead:

```javascript
/**
 * @param {string} sellerId
 * @param {{ sku: string, qty: number, rate: number }[]} items
 * @returns {{ subtotal: number, gst_total: number, grand_total: number }}
 */
function calculateTransaction(sellerId, items) { ... }
```

### Migration Plan
1. Convert `tsconfig.json` to allow JS files (`allowJs: true`, `checkJs: false`)
2. Rename new feature files as `.jsx` / `.js` — do not bulk-rename existing files
3. Keep `next.config.ts`, `tailwind.config.ts`, and config files as TypeScript (they are static, low-maintenance)
4. Supabase client wrapper remains `.ts` — export typed helpers for use in JS files
5. `/packages/accounting` remains fully TypeScript

### Reversibility
This decision is reversible. When the feature set stabilizes and human developers join, files can be migrated to TypeScript incrementally. JSDoc annotations become the migration guide.

### Trade-offs Accepted
| Accepted Risk | Mitigation |
|---------------|-----------|
| No compile-time type checking on frontend | JSDoc + ESLint for basic safety |
| Runtime type errors possible | Input validation at system boundaries (Zod in accounting package only) |
| Less self-documenting contracts | JSDoc annotations required on all exported functions |
