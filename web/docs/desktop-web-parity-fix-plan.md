# Desktop ↔ Web — Logic & Flow Parity Fix Plan

**Date**: 2026-05-31
**Author**: engineering review
**Scope**: Business-logic / data-flow **correctness and sync parity** between the PocketBase-backed Electron desktop terminal (`/desktop`) and the Supabase-backed web app (`/web`).
**Relationship to existing doc**: `web/docs/gap-analysis-web-vs-desktop.md` tracks *feature presence* (what exists where). **This** doc tracks *logic correctness, flow semantics, and the cross-platform data contract* — i.e. where the same feature exists on both sides but **behaves differently** or will **break cloud sync**. It supersedes several stale claims in that doc (see §2).

---

## 1. TL;DR

The two apps share UI lineage and an **identical per-line GST model** (5% tax-exclusive, `toFixed(2)`), but their business-logic backends have diverged. Desktop orchestrates everything **client-side straight to PocketBase** with no enforcement layer; web pushes integrity into **Postgres triggers**. Three classes of mismatch result:

- **(A) Desktop correctness/integrity bugs** — credit limit never enforced, refund/cancel double-apply, partial-refund skips khata, shift reconciliation mis-counts refunds, non-atomic checkout.
- **(B) Sync-contract divergences** — payment-method casing/values, order status enum, signature payload, order-number scheme, tax-exempt — any of which makes desktop→Supabase order sync fail or corrupt.
- **(C) Architecture/flow divergences** — no multi-tenancy on desktop, cosmetic role gating, open PocketBase access rules, no audit trail, single-step vs approval refunds.

**Important caveat from the web review:** several behaviors we'd want desktop to "match" are **also broken on web** (client-trusted POS totals & signature, non-race-safe order numbers, shift tracking that silently never runs). So the goal is **converge both apps onto a corrected canonical contract (§4)** — not copy web verbatim. A few fixes must land on **web first** (§6) so desktop aligns to a correct target.

## 4. Canonical cross-platform contract (the target both apps converge to)

This is the **single source of truth** both the PocketBase schema/desktop hooks and the Supabase schema/web routes must implement identically. Where web and desktop currently differ, the **Canonical** column wins.

| Concern | Canonical rule | Current desktop | Current web |
|---|---|---|---|
| **GST** | Flat **5%, tax-exclusive**; one shared helper used everywhere; discount per-unit pre-tax | `lib/gst.ts` (matches) | inline-duplicated; **plus a contradictory inclusive model in product forms** (`product-form.jsx:268`) — fix on web |
| **Money rounding** | Round **per line** to 2 dp, then **sum**; never round the aggregate separately | rounds aggregate (`gst.ts:32-33`) — change | per-line sum (matches) |
| **Order number** | Server/DB-issued, **gapless, scoped to terminal+seller+year**, e.g. `{TERMINAL}-{YYYY}-{SEQ}`; offline terminals carry a unique terminal id so numbers never collide on sync | client count+1, collides cross-terminal | client `Math.random()` — fix on web |
| **Digital signature** | `SHA-256( orderNo : grandTotal : seller_tpn )` — **no timestamp**, recomputable from the stored row; verified server/DB-side on sync | adds `:timestamp` (unverifiable) — change | matches payload, but never verified — add verify on sync |
| **Payment method** | UPPERCASE enum **`CASH` \| `CREDIT` \| `ONLINE`**; desktop's mBoB/mPay/RTGS map to `ONLINE` with a `payment_channel` sub-field if the distinction must be kept | lowercase 5-value — map | `CASH/CREDIT/ONLINE` (canonical) |
| **Order status** | Shared lifecycle enum; desktop may use a subset but values/casing must be exact (`CONFIRMED`, `CANCELLED`, `REFUND_REQUESTED`, `REFUND_APPROVED`, `REFUNDED`, …) | 3-value `CONFIRMED/CANCELLED/REFUNDED` — extend casing-compatible | 19-value (canonical) |
| **Credit limit** | Enforced **atomically at the data layer** (DB trigger / PocketBase hook) on credit-sale confirm: reject if `balance + grand_total > credit_limit`; app-layer check is advisory UX only | none — add | DB trigger (canonical pattern) |
| **Refund** | Guarded by current status (idempotent); restores stock **and** reverses khata proportionally for credit sales; partial refunds reverse a proportional khata amount | partial skips khata; no guard — fix | refund never reverses khata — fix on web |
| **Cancel** | Guarded by current status; restores stock + reverses khata; locate khata account by **order reference**, not customer-name string | name-match (`use-orders.ts:50-55`) — change | order-ref trigger (canonical) |
| **Shift ↔ order link** | Each sale/refund writes a **shift_transaction row in the same transaction** as the order; reconciliation reads those rows; expected cash = `opening_float + Σ CASH sales − Σ CASH refunds + Σ cash-in − Σ cash-out` | timestamp scan; refunds counted regardless of method — fix | track call silently fails — fix on web |
| **Tax-exempt** | Decide canonical: either support on both (add `tax_exempt` boolean to web orders) or remove from desktop. If kept, per-item `total` must be recomputed (not GST-inclusive) when exempt | desktop only; item `total` stays GST-inclusive when exempt (`use-checkout.ts:82-92`) — bug | absent |
| **Tenancy** | Every synced row carries `entity_id`; desktop stamps its store's entity id on orders/movements/khata before sync | not set on movements | `entity_id`-scoped |

> **Trigger safety (2026-05-31, Migration 074) — the sync blocker, resolved:** synced orders carry `orders.origin = 'TERMINAL_SYNC'` (default `'CLOUD'`); the three cloud confirm-time triggers — `orders_guard_stock`, `orders_deduct_stock`, `orders_khata_debit` — were recreated with a `WHEN (NEW.origin IS DISTINCT FROM 'TERMINAL_SYNC')` clause, so a synced order does **not** re-deduct cloud stock, re-debit khata, or get **rejected** by the stock guard (it's an already-completed offline sale). The trigger **function bodies are untouched** — only the trigger definitions changed. **Stock stays correct** because `inventory_movements` is the single source of truth: cloud-rung orders get their SALE movements from `deduct_stock_on_confirm`; synced orders bring their own movements (which apply via `apply_inventory_movement`) and the order trigger is simply skipped. **Remaining khata follow-up:** the cloud khata *balance* is currently driven by `khata_debit_on_confirm` (now skipped for synced orders), so synced **credit** sales won't move the cloud balance yet — reconciling it from the synced `khata_transactions` (a txn-driven balance trigger, or the sync worker applying the delta) is the open decision. (Note `orders_khata_debit` is `AFTER UPDATE` only, so an INSERTed synced order never fired it anyway — the guard is defensive + explicit.) **Verified:** the WHEN-guard mechanism on throwaway Postgres — CLOUD/default orders fire the trigger, `TERMINAL_SYNC` orders skip it; the `origin` default + CHECK backfill works. **Not** runtime-verified against the full cloud schema (no local Supabase — apply Migration 074 there).

> **Khata-balance reconciliation (2026-05-31, Migration 075) — resolved:** synced credit-sale DEBITs and repayment/reversal CREDITs move the cloud balance via **idempotent delta-application**, not a recompute. *Why not* a flat `balance = SUM(txns)` recompute: the cloud floors per-operation (`GREATEST(0, balance − x)` on CREDIT/repayment), so the balance is **path-dependent** — a flat sum would *alter* correct cloud-originated balances. Instead: `khata_transactions.external_id` (= `"<machine_id>:<terminal_txn_id>"`, unique) makes each terminal txn claimable exactly once, and the `apply_synced_khata_txn(account, external_id, type, amount, …)` RPC moves the balance with the **same semantics as the cloud triggers** (DEBIT `+amount`; CREDIT `GREATEST(0, balance − amount)`) ONLY on a fresh claim — a re-sync returns `'duplicate'` and is a no-op. The sync worker (`syncKhata` in `register-order-sync.ts`) matches each terminal khata account to a cloud account by `(creditor_entity_id, debtor_phone, CONSUMER)`, creating it if absent **without** overwriting the cloud's `credit_limit` (cloud is authoritative), then replays its txns. **Verified** on throwaway Postgres AND live on the local Supabase stack (2026-06-06, after `db reset`): DEBIT adds, CREDIT subtracts with floor, a re-applied `external_id` is a no-op, `balance_after` backfilled. **Bug the real schema caught + fixed:** `khata_transactions.created_by` is NOT NULL (FK → `user_profiles`), but `syncKhata` calls the RPC *without* `p_created_by`, so the insert hit the NOT-NULL constraint (the throwaway-PG stub didn't have it, so it slipped through). Fix: `apply_synced_khata_txn` now resolves an actor when the caller passes none — a `user_profiles` of the account's entity, preferring OWNER → ADMIN → MANAGER, mirroring how `khata_debit_on_confirm` attributes a txn (raises a clear error if the entity somehow has no user). Re-verified live calling it exactly as `syncKhata` does (no `p_created_by`): applied/duplicate/floor all correct, actor auto-resolved. **Actor attribution (2026-06-06, done):** the ingest now selects `terminal_tokens.created_by` and threads it as `batch.actorId` → `syncTerminalBatch` → `syncKhata(…, actorId)` → `apply_synced_khata_txn(p_created_by)`, so a synced khata txn is attributed to the **admin who provisioned the terminal**; the entity-owner resolution stays as the safety-net fallback when a token has no issuer. Verified live: passing an issuer attributes the txn to *that* user (not the fallback); passing none falls back to the owner. package + web tsc clean; ingest route eslint clean. **Done (2026-06-06):** `reference_id` → cloud `order_id` remap (threaded via the syncOrders order-id map → `p_order_id`); null-phone account matching (a phoneless account matches by `null phone + name` instead of spawning duplicates); `order_items` sync (Migration 079).
